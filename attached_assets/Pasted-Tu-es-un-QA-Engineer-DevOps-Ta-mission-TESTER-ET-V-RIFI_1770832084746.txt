Tu es un QA Engineer + DevOps. Ta mission: TESTER ET VÉRIFIER FitMatch.fr de bout en bout en environnement PROD (mode réel) et en mode DEV si nécessaire.

Objectif:
- Tester le parcours complet côté CLIENT et côté COACH
- Vérifier toutes les règles métier (abonnement actif/inactif, 404, paiement, emails, multi-langues)
- Détecter bugs, failles, incohérences UI/UX, erreurs serveur/logs
- Livrer un rapport final clair + une checklist “OK/KO” + corrections à faire

⚠️ Règles:
- Ne suppose rien: tu vérifies réellement
- Tu testes sur desktop + mobile (responsive)
- Tu vérifies les logs et erreurs runtime (console + serveur)
- Tu testes les cas normaux ET les cas limites (edge cases)

1) Setup & Santé du projet
- Vérifie que le site charge vite, pas d’erreurs console, pas d’erreurs 500
- Vérifie les variables d’environnement critiques (STRIPE, RESEND, DB, BASE_URL, etc.)
- Vérifie le mode (démo vs réel) et confirme que le paiement réel est bien activé en prod
- Vérifie SSL, redirections http->https, pages 404/500 propres

2) Parcours COACH (de A à Z)
A. Inscription / Connexion coach
- Créer un compte coach, tester email de bienvenue si existant
- Connexion / déconnexion / reset password (si dispo)
- Vérifier les validations (email invalide, mot de passe faible)

B. Profil public du coach
- Remplir profil (nom, bio, tarifs, disponibilités, langue, photo, etc.)
- Vérifier l’affichage public (page profil + booking form)
- Tester URLs: profil existe / profil supprimé / profil incomplet

C. Abonnement coach (règle business CRITIQUE)
- Cas 1: abonnement ACTIF => profil public accessible + réservations possibles
- Cas 2: abonnement INACTIF (blocked/cancelled/past_due) => profil public et booking form doivent renvoyer 404
- Vérifie que la restriction est appliquée partout (UI + API + routes)

D. Paiements coach (Stripe)
- Vérifie que Stripe est en mode LIVE si prod
- Simule: paiement réussi, paiement échoué, carte refusée
- Vérifie que le statut d’abonnement se met à jour correctement
- Vérifie emails: succès, échec, blocked, restored (si implémentés)

3) Parcours CLIENT (de A à Z)
A. Découverte
- Trouver un coach (recherche, filtres si existants)
- Ouvrir le profil public + voir prix + disponibilités

B. Réservation
- Réserver une séance sur un créneau valide
- Tester un créneau déjà pris (doit refuser proprement)
- Tester plusieurs réservations successives
- Vérifier confirmations UI + création correcte en base

C. Paiement client (si paiement côté client existe)
- Tester paiement Stripe: succès / échec / annulation / retry
- Vérifier reçu/receipt email
- Vérifier que la séance passe au bon statut (paid/confirmed)

D. Emails & Notifications
- Vérifier tous les emails (client + coach):
  - booking confirmation
  - booking cancellation (client + coach)
  - reminders 24h et 2h avant (si scheduler)
  - coach notification new request
  - receipt paiement
- Vérifier que la langue correspond au user (FR/EN/ES/DE/IT/PT/AR)
- Vérifier sujets + contenus + liens (base_url correct) + pas de variables manquantes

4) Annulation / Remboursement / Conflits
- Annuler côté client (si possible) => statut + email client + email coach
- Annuler côté coach (si possible) => idem
- Tester annulation à moins de X heures (si règle)
- Vérifier comportement des paiements (refund ou non selon règle)

5) Sécurité / Qualité
- Vérifier que personne ne peut accéder aux dashboards coach/client sans auth
- Vérifier qu’un client ne peut pas réserver via API un coach “inactif”
- Vérifier protections basiques: rate limit, validation backend, injection, accès aux IDs
- Vérifier que les secrets ne leakent pas côté client (Stripe secret key jamais exposée)
- Vérifier CORS, cookies, sessions, headers de sécurité

6) Performance & UX
- Temps de chargement pages principales
- Mobile: boutons cliquables, formulaires utilisables, pages booking clean
- Accessibilité basique: champs labels, erreurs lisibles

7) Rapport FINAL à livrer
Dans un rapport markdown, fournis:
- Résumé (ce qui marche / ce qui bloque)
- Checklist OK/KO par module (Coach, Client, Stripe, Emails, Abonnements, 404)
- Liste des bugs classés: CRITICAL / HIGH / MEDIUM / LOW
- Pour chaque bug: étapes reproduction, comportement attendu, comportement réel, capture console/log si possible, proposition de fix
- Recommandations “avant lancement officiel” (5-10 points max)

Commence par exécuter les tests dans l’ordre, et STOP immédiatement si tu trouves un bug CRITICAL (paiement live faux, 404 non respecté, secrets exposés). Ensuite tu continues après avoir proposé un fix.
